Node学习（五）- 内存控制 - V8的垃圾回收与内存机制

发表于 2017-03-11 | 分类于 技术笔记

在Node中通过JavaScript使用内存时会发现只能使用部分内存（64位系统下约为1.4GB，32位系统性约为0.7GB）。这是因为Node基于V8构建，所以Node中JavaScript对象的分配都是通过V8自己的方式进行分配和管理的。

V8的对象分配

在V8中，所有的JavaScript对象都通过堆进行分配。如果已申请的堆空间内存不够分配，就继续申请堆内存直到堆的大小超过V8的限制为止。

那么，为什么V8要限制堆的大小呢？
表层原因为V8最初为浏览器设计，不太可能遇到用大量内存的场景，对于网页来说V8的限制值已经足够。深层原因则是V8的垃圾回收机制的限制。
按官方说法，以1.5GB的垃圾回收堆内存为例，V8做一次小的垃圾回收需要50毫秒以上，做一次非增量式垃圾回收要1秒以上。这是垃圾回收引起JavaScript线程暂停执行的时间，在这种情况下应用的性能和响应能力都会直线下降。

不过，V8提供了让用户来调整内存限制的大小的入口，通过修改以下Node启动参数，这两个参数的意思稍后会介绍：

--max-old-space-size // 单位为MB
--max-new-space-size // 单位为KB

V8的垃圾回收机制

V8的垃圾回收策略基于分代式垃圾回收机制：根据对象生命周期（存活时间）的长短进行不同的分代，针对不同的分代采取不同的垃圾回收算法。

在V8中，主要将内存分为新生代和老生代：

  • 新生代：存活时间短，可以通过–max-new-space-size设置最大值（64位系统默认32MB，32位默认16MB）
  • 老生代：存活时间长，常驻内存，可以通过–max-old-space-size设置最大值（64位系统默认1400MB，32位默认700MB）

Scavenge算法

在分代的基础上，新生代中的对象主要通过Scavenge算法进行垃圾回收。Scavenge算法的具体实现中，采用的Cheney算法。

Cheney算法将堆内存一分为二，每一部分成为semispace。这两个semispace，只有一个处于使用中，称为From空间；另一个处于闲置状态，称为To空间。

  • 我们分配对象时，先在From空间进行分配
  • 当垃圾回收时，检查From空间中的存活对象，并将其复制到To空间中，非存活对象占用空间将被释放
  • 复制完成后，From空间与To空间角色互换

Scavenge算法的缺点：只能使用堆内存的一半。但对于生命周期短的对象而言，存活对象总是只占所有对象的少部分，所以时间效率表现优异（以空间换时间，但也因此无法大规模应用到所有的垃圾回收场景中）。

晋升：当一个对象被多次复制还存活，则被认为是长生命周期对象——将从新生代中移动到老生代中。这个过程被称为晋升。
对象的晋升条件主要有两个：

  • 是否经历过Scavenge回收（V8在对象从From空间挪往To空间时会检查它的内存地址来判断这个对象是否已经历过一次Scavenge回收，如果有，那么它将晋升）
  • To空间的内存占比是否超过限制（25%，如果占比过高会影响后续内存分配）

Mark-Sweep & Mark-Compact

对于老生代对象，如果采用Scavenge算法会有两个问题

  • 存活对象较多，复制存活对象效率低
  • 浪费一般空间
    因此，V8中针对老生代对象，采用Mark-Sweep和Mark-Compact结合的方式

Mark-Sweep

分为标记和清楚两个阶段：

  • 标记阶段：遍历堆中所有对象，并标记活着的对象
  • 清楚：只清楚没有被标记的对象

Mark-Sweep的缺点：在一次清楚后，内存空间不连续，遇到大对象碎片空间无法完成此次分配，会提前触发垃圾回收，而这次回收本是不必要的。

Mark-Compact

为了解决Mark-Sweep的碎片问题而被提出。
Mark-Compact由Mark-Sweep演变而来，对象在标记死亡后，整理过程中将活着的对象往一端移动，移动完成后，直接清理掉边界外的内存。

Mark-Sweep和Mark-Compact在V8中是结合使用的，由于Mark-Compact需要移动对象，所以它的执行速度不可能很快。所以V8主要使用Mark-Sweep，在空间不足以对从新生代中晋升过来的对象进行分配时才使用Mark-Compact。

Incremental Marking（增量标记）

为了避免出现JavaScript应用逻辑与垃圾回收器看到的不一致的情况，垃圾回收的3种基本算法都需要将应用逻辑暂停下来，待执行完垃圾回收后再恢复执行，这种行为被称为“全停顿”（stop-the-world）
为了降低全堆垃圾回收带来的停顿时间，将标记动作改为增量标记，也就是拆为“步进”，与应用逻辑交替执行直到标记阶段完成。


V8在经过增量标记改进后，垃圾回收的最大停顿时间可以减少到原本的1/6左右

lazy sweeping（延迟清理）

增量标记完成后，延迟清理开始。此时所有对象存活状态都已被标记，堆准确知道可以回收多少内存，然而此时不必一次全部回收死亡对象，可以采用延迟清理的处理手段，垃圾回收器根据需要来选择回收部分内存，直到全部垃圾对象回收完毕，整个增量标记-延期清理的周期结束，重新开始标记。
